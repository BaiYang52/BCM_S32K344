## BCM_S32K344 / Autosar APP 工程搭建

为后续 Autosar 工程做底层裸机骨架准备（NXP S32K344 + IAR）。

### 已完成的修改

1. **`_SourceCode/StartUp/startup_ARMCM7.c` —— 修复 Reset_Handler**
   - 修复了原文件中文注释乱码（GBK 被 Latin-1 解码）。
   - **关键 bug**：原文件中"备用代码"被混在 `/* ... */` 块注释里，但行内嵌了 `/* SWT_0->SR */` 等段注释——这会让 IAR 试图编译这 3 行裸地址写入代码（因为外层 `/* */` 在遇到行内 `/* */` 时提前结束）。已彻底改写：直接用 `(volatile uint32_t *)0x402A0010U` 写 SR、CR 寄存器，关闭 SWT_0。
   - 代码放在 Reset_Handler 进入后第一行，确保在 `__PROGRAM_START()`（BSS 清零 + DATA 拷贝）之前。

2. **`_SourceCode/Link/S32K344_flash.icf` —— 为 FBL 预留空间**
   - PFLASH：起始地址 0x400000 → **0x440000**（预留 256KB 给 FBL），size 0x3D4000 → 0x394000。
   - SRAM：起始地址 0x20408000 → **0x20418000**（预留 64KB 给 FBL 热复位标记 + 内核栈），size 0x48000 → 0x38000。
   - APP IVT 起始 = 0x440400（boot_header 之后 0x400 偏移），CM7_START_ADDRESS 自动跟着挪过去。
   - PFLASH end = 0x7D3FFF，与 BAF 起始 0x7F4000 留有 128KB 安全间隔。

3. **`BCM_APP.ewp` —— 补全文件组**
   - 工程组里原本只有 9 个文件，缺 `system_ARMCM7.c`（提供 SystemInit 实现，否则链接报 undefined reference）。已补上 system_ARMCM7.c/h、cmsis_compiler.h、cmsis_version.h、mpu_armv7.h。
   - 工程宏定义已含 `ARMCM7_DP` 和 `START_FROM_FLASH`，符合 .icf 里的 `#ifdef START_FROM_FLASH` 分支。

### 第二轮：修复 BCM_APP.ewp 的 CCDiag 字段污染

**问题症状**：用户截图显示 4 个 `Ms001: diagnostic tag "E:\Project\GitHub\...\StartUp" does not exist` 错误，分布在 `main.c / boot_header.c / startup_ARMCM7.c / system_ARMCM7.c`。

**根因**：IAR 把 `CCDiagSuppress/Remark/Warning/Error` 的 `<state>` 字段当作 **诊断消息 ID** 来解析，但 .ewp 文件里被填入了**目录路径**或**文件名占位符**：
- Debug 配置的 `CCDiagSuppress` = `$PROJ_DIR$\_SourceCode\StartUp`（目录路径）
- Release 配置更乱：`CCDiagSuppress = $FILE_BNAME$.o`、`CCDiagRemark = $PROJ_DIR$\_SourceCode\StartUp`、`CCDiagWarning` 被错误地填了 `ARMCM7_DP / START_FROM_FLASH`（本应在 `CCDefines`）、`CCDiagError = $FILE_BNAME$.o`

**修复**：四个 `CCDiag*` 字段全部清空（`<state></state>`），IAR 默认就是不禁用任何诊断。同时校验 `CCDefines` 仍正确：Debug = `ARMCM7_DP + START_FROM_FLASH`，Release = `NDEBUG + ARMCM7_DP + START_FROM_FLASH`。

**注意**：这是 .ewp 文件本身的字段错配（很可能是从别的项目复制粘贴时残留），与代码本身无关。修复后 `main.c / boot_header.c / startup_ARMCM7.c / system_ARMCM7.c` 这 4 个文件编译时应不再报这个错。

### 第三轮：修复 cmsis_compiler.h include 形式

**问题症状**：`Fatal Error[Pe1696]: cannot open source file "cmsis_iccarm.h"`（在 `cmsis_compiler.h:79`）。

**根因**：第 79 行 IAR 分支写的是 `#include <cmsis_iccarm.h>`（尖括号），IAR 会去系统 include 路径找——而 `cmsis_iccarm.h` 实际就在 `_SourceCode\StartUp\` 目录里（IAR 标准的 CMSIS 头），不靠系统路径。

**修复**：改成 `#include "cmsis_iccarm.h"`（双引号），编译器会先在当前文件所在目录查找，直接命中。

### 第四轮：用户修正内存布局策略

**用户后续规划**（总 4MB Flash 划分）：
| 段 | 大小 | 用途 |
|---|---|---|
| Bootloader | 256KB | FBL |
| Boot Config | 64KB | Boot Flag、安全配置 |
| APP A | 1408KB | 主程序（**当前工程**） |
| APP B | 1408KB | OTA 升级备用 |
| Calibration | 256KB | XCP 标定 |
| Fee/NvM | 512KB | AUTOSAR NvM |
| Reserved | 剩余 | 后续扩展 |

合计 256+64+1408+1408+256+512 = 3904KB = 0x3F0000（约 3.81MB），留约 196KB 后续扩展。

**用户决定**：现阶段**不预先平移 APP 地址**。理由是：
1. FBL、标定等通过工具做（量产烧录器），不在当前工程内集成。
2. 当前只关心 **standalone build → 烧录 → debug** 能跑通即可。
3. 未来切换方案：右击 `boot_header.c` → Exclude from Build + 改 .icf 起始地址。

**对 .icf 的调整**：
- `__region_PFLASH_start__` 改回 `0x400000`（不是 0x440000）
- `__size_pflash__` 改回 `0x3F4000`（不是 0x394000），APP 占满 0x400000~0x7F3FFF
- `__region_SRAM_start__` 改回 `0x20408000`
- `__size_sram__` 改回 `0x48000`
- 在 .icf 头部加了明确的"FBL 切换说明"注释，告诉未来自己/同事：FBL 准备好后改这 4 个值 + Exclude boot_header.c 即可。

**结果**：当前 .icf = 原始 NXP 默认布局（最大可用 Flash + SRAM），方便 debug 阶段跑通全部内存。所有"预留 FBL"改动**已经撤销**，避免现阶段 debug 阶段出现奇奇怪怪的边界问题。

**未来 FBL 切换的 4 行改动**（按用户 256KB FBL + 64KB Boot Config 划分）：
- `__region_PFLASH_start__` = 0x440000（256KB + 64KB = 0x40000 偏移）
- `__size_pflash__` = 0x3F4000 - 0x40000 = 0x3B4000
- `__region_SRAM_start__` = 0x20418000（保留前 64KB 给 FBL）
- `__size_sram__` = 0x48000 - 0x10000 = 0x38000
