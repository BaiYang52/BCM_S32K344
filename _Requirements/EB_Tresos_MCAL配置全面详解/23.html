第23章：错误检测与报告（Det）——Det模块概述、错误ID定义、开发模式与生产模式、日志输出

各位同学，今天我们来聊聊AUTOSAR里一个特别“低调”但极其重要的模块——Det（Default Error Tracer）。说白了，它就是整个系统的“黑匣子”。

我在做第一个量产项目时，就吃过Det配置不到位的亏。当时车机跑着跑着突然死机，日志里啥也没留下，排查了整整两周。后来才发现，是Det模块在生产模式下被完全关闭了。嗯，从那以后，我对Det的敬畏心就上来了。

23.1 Det模块概述

Det的全称是Default Error Tracer，翻译过来就是“默认错误追踪器”。它的职责很简单：接收并记录来自MCAL、BSW甚至RTE层上报的错误信息。

你想想看，一个复杂的汽车电子系统，可能有上百个模块在同时运行。任何一个模块出了错，如果没有统一的错误上报机制，那调试起来简直就是噩梦。Det就是那个“统一收口”的地方。

我个人习惯把Det理解成一个“错误路由器”。它不负责处理错误，只负责把错误信息记录下来，或者转发给其他模块（比如DET）。

核心要点：Det模块本身不产生错误，它只是错误信息的“搬运工”。所有MCAL模块（如GPT、ICU、SPI等）在检测到异常时，都会调用Det_ReportError()函数来上报。

23.2 错误ID定义

在AUTOSAR里，每个错误都有一个唯一的ID。这个ID由三部分组成：

  • 模块ID（ModuleId）：标识是哪个模块报的错。比如GPT模块的ID是0x0A，ICU是0x0B。
  • 实例ID（InstanceId）：如果模块有多个实例，用这个来区分。比如你有两个GPT定时器，实例0和实例1。
  • API服务ID（ApiId）：标识是哪个API函数检测到了错误。比如Gpt_StartTimer()的ApiId是0x01。

这三者组合起来，就能精确定位到“哪个模块的哪个实例的哪个函数”出了问题。

我的经验：在项目初期，一定要和系统架构师确认好模块ID的分配。我曾经见过两个团队用了相同的模块ID，结果错误日志完全乱套了。后来我们统一在EB Tresos里维护了一个“模块ID分配表”，才彻底解决了这个问题。

错误ID的定义通常在Det_Cfg.h中完成，示例如下：

/* 模块ID定义 */
#define DET_MODULE_ID_GPT         0x0A
#define DET_MODULE_ID_ICU         0x0B
#define DET_MODULE_ID_SPI         0x0C

/* API服务ID定义（以GPT为例） */
#define GPT_API_ID_START_TIMER    0x01
#define GPT_API_ID_STOP_TIMER     0x02
#define GPT_API_ID_GET_TIME_ELAPSED 0x03

23.3 开发模式与生产模式

这是Det配置里最关键的决策点。说白了，就是“调试时怎么报错”和“量产时怎么报错”的区别。

开发模式（Development Mode）

在开发阶段，我们希望错误信息越详细越好。Det在开发模式下会：

  • 记录完整的错误ID（模块ID + 实例ID + API服务ID）
  • 输出详细的错误描述字符串
  • 触发断点或回调函数，方便调试器捕获
  • 记录错误发生的时间戳

我记得有一次，客户反馈某个ECU在高温测试时偶尔会重启。我们打开Det的开发模式，跑了三天三夜的高温循环，终于从日志里发现是SPI通信超时导致的。如果没有详细的错误ID和时间戳，这种偶发问题根本抓不到。

生产模式（Production Mode）

量产时，情况就完全不同了。你想想看，一辆车上有几十个ECU，每个ECU每秒可能产生成千上万条错误日志。如果全部记录下来，存储空间根本不够用。

生产模式下，Det通常会：

  • 只记录错误ID，不记录详细描述
  • 限制日志条数（比如最多存100条）
  • 采用环形缓冲区，新日志覆盖旧日志
  • 关闭断点和回调函数

重要提醒：生产模式下千万不要把Det完全关闭！我曾经见过一个项目，为了节省资源把Det彻底禁用了。结果车辆在路试时出现偶发故障，售后完全无法定位问题。最后不得不给所有车辆刷固件，重新开启Det。这个教训太深刻了。

在EB Tresos中，开发模式和生产模式的切换通常通过一个宏定义来控制：

/* 在Det_Cfg.h中 */
#define DET_DEV_MODE              STD_ON   /* 开发模式：ON；生产模式：OFF */

#if (DET_DEV_MODE == STD_ON)
  #define DET_REPORT_ERROR(ModuleId, InstanceId, ApiId, ErrorCode) \
    Det_ReportError(ModuleId, InstanceId, ApiId, ErrorCode)
#else
  #define DET_REPORT_ERROR(ModuleId, InstanceId, ApiId, ErrorCode) \
    /* 生产模式下，只记录错误计数，不输出详细日志 */
    Det_IncrementErrorCounter(ModuleId, InstanceId, ApiId)
#endif

23.4 日志输出

Det的日志输出方式，取决于你的硬件平台和调试工具。常见的输出方式有：

输出方式 适用场景 优点 缺点
串口输出（UART） 开发调试 实时查看，简单可靠 需要硬件串口线
RAM缓冲区 生产模式 速度快，不占用额外硬件 掉电丢失，容量有限
NVM存储 售后诊断 掉电不丢失 写入速度慢，有寿命限制
CAN/LIN总线 整车网络 可远程监控 占用总线带宽

我个人最常用的组合是：开发阶段用串口输出，量产阶段用RAM缓冲区 + NVM存储。这样既能保证调试效率，又能确保售后可追溯。

日志输出的格式通常如下：

[Det] ModuleId=0x0A, InstanceId=0x00, ApiId=0x01, ErrorCode=0x03
[Det] Timestamp: 12345678 us
[Det] Description: GPT timer start failed - invalid channel ID

小技巧：在开发模式下，我习惯在日志里加上一个“错误等级”字段。比如FATAL、ERROR、WARNING、INFO。这样在查看海量日志时，可以快速过滤出真正需要关注的致命错误。这个字段在AUTOSAR标准里是可选的，但强烈建议加上。

23.5 避坑指南

最后，分享几个我在项目中踩过的坑：

  • 不要在生产模式下完全关闭Det——哪怕只保留一个错误计数器，也比什么都没有强。
  • 注意日志缓冲区的大小——太小会丢失关键信息，太大会浪费RAM。我一般按“最大错误频率 × 10秒”来估算。
  • 开发模式和生产模式的切换要自动化——不要手动改代码，用编译选项或预处理器宏来控制。否则很容易出现“发布版本忘了关开发模式”的尴尬。
  • 错误ID的命名要有规律——比如所有GPT的错误都以GPT_开头，所有ICU的错误都以ICU_开头。这样在日志里一眼就能看出是哪个模块的问题。

好了，关于Det模块的核心内容就讲到这里。下一章我们会深入讲解Det的配置项，包括如何通过EB Tresos来配置错误缓冲区大小、输出接口选择等。到时候我会带大家手把手操作一遍。

记住一句话：Det配置得好，调试没烦恼；Det配置得差，加班到天亮。
