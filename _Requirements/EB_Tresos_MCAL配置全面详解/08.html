第8章 通用输入输出（Gpt）：定时器的灵魂与脉搏

各位同学，今天我们聊聊Gpt模块。说实话，在AUTOSAR的MCAL层里，Gpt算是比较「温柔」的一个模块。它不像Can或者Lin那样要处理复杂的协议栈，也不像Adc那样要跟模拟信号打交道。但你别小看它——定时器是整个嵌入式系统的「心跳」，没有它，系统就没了节奏感。

我个人习惯把Gpt比作一个「闹钟」。你设定好时间，到点了它就响。只不过在汽车里，这个闹钟要精确到微秒级，而且可能同时有几十个闹钟在跑。嗯，这就是我们要面对的现实。

8.1 Gpt模块原理：定时器到底在干什么？

Gpt模块的核心，说白了就是一个计数器。它内部有一个寄存器，每个时钟周期加1。当这个值达到你设定的目标值时，就产生一个事件。就这么简单。

但为什么说它复杂呢？因为汽车里有太多「定时」的需求：

  • 轮询某个传感器，每10ms读一次
  • 发送CAN报文，每100ms发一帧
  • 执行某个控制算法，每1ms算一次
  • 等待某个硬件稳定，延迟500μs再采样

这些需求，全都要靠Gpt模块来支撑。我在项目中遇到过最夸张的一次，一个ECU里同时用了8个定时器通道，每个通道的周期都不一样。你想想看，这要是配置错了，整个系统就乱套了。

核心概念：Gpt模块本质上是一个硬件定时器的软件抽象层。它把硬件定时器的寄存器操作封装成了标准API，让上层应用不用关心底层用的是哪个定时器、什么型号的芯片。

8.2 定时器通道配置：每个通道都是一个独立闹钟

在EB Tresos里配置Gpt通道，其实就是在告诉工具：我要用哪个定时器、什么模式、多长时间触发一次。我建议你先把硬件手册翻出来，看看芯片上有几个定时器模块，每个模块有几个通道。

配置时主要关注这几个参数：

参数名称 说明 我的建议
GptChannelId 通道ID，唯一标识 命名要有规律，比如GptCh_10ms
GptChannelMode 单次模式或周期模式 看需求，后面细讲
GptChannelTickFrequency 时钟频率，决定精度 别设太高，功耗会上去
GptChannelTickValueMax 最大计数值 注意溢出问题
GptNotification 是否使能中断 大多数情况都要使能

我记得有一次，一个同事把GptChannelTickFrequency配成了系统时钟的1倍。结果定时器每1μs就触发一次中断，CPU全被中断占满了，主循环根本跑不起来。嗯，这就是典型的「配置过度」。

小技巧：配置定时器频率时，先算一下你需要的精度。比如你要10ms的定时，用1MHz的时钟就够了，没必要用100MHz。精度够用就行，别浪费资源。

8.3 中断与回调函数：定时器到点了怎么办？

定时器到点了，系统得知道。有两种方式：轮询和中断。在汽车里，我们几乎都用中断。为什么？因为轮询太浪费CPU了，你想想看，主循环里每跑一圈都要去查一下定时器有没有到点，多累啊。

中断的流程是这样的：

  1. 定时器计数值达到目标值
  2. 硬件触发中断
  3. CPU暂停当前任务，跳转到中断服务函数
  4. 在中断里调用你注册的回调函数
  5. 执行完回调，返回原来的任务

在EB Tresos里配置回调函数，其实就是把函数名填到配置项里。比如：

/* 在代码中定义回调函数 */
void MyGptCallback_10ms(void)
{
    /* 每10ms执行一次的任务 */
    Can_Write(&CanTxMsg);
}

/* 在EB Tresos中配置 */
GptNotification = MyGptCallback_10ms

这里有个坑，我必须要说。回调函数是在中断上下文里执行的，所以它必须短小精悍。我曾经见过有人在回调函数里调用了printf，结果系统直接卡死。为什么？因为printf太慢了，而且可能涉及系统调用，在中断里根本不能跑。

警告：回调函数里不要做耗时操作！不要调用printf、malloc、或者任何可能阻塞的函数。原则是：能多短就多短，最好只设置一个标志位，让主循环去处理。

8.4 单次与周期模式：一次还是反复？

这是Gpt模块最基础的选择题。单次模式，就是定时器只跑一次，到点触发后自动停止。周期模式，就是定时器反复跑，每次到点都触发，周而复始。

什么时候用单次？举个例子：你要发送一个诊断请求，需要等待ECU回复。如果500ms内没收到回复，就认为超时。这时候你就启动一个单次定时器，500ms后触发超时处理。

什么时候用周期？比如你要每10ms采集一次传感器数据，那就用周期模式。定时器会一直跑，每10ms触发一次中断，你在回调里读取传感器值。

在EB Tresos里配置很简单：

/* 单次模式配置 */
GptChannelMode = GPT_CH_MODE_ONESHOT

/* 周期模式配置 */
GptChannelMode = GPT_CH_MODE_CONTINUOUS

我个人习惯是：能用周期就不用单次。为什么？因为单次模式每次触发后都要重新启动，多了一步操作。而且如果忘记重新启动，定时器就停了，系统可能出问题。周期模式就省心多了，配置好就不用管了。

但也不是绝对的。我记得有一个项目，需要做一次性的延时等待。我用周期模式配了个定时器，结果每次触发后都要手动停止，代码写得很别扭。后来改成单次模式，清爽多了。所以，看场景选模式，别死板。

经验总结：

  • 需要反复执行的任务 → 周期模式
  • 只需要执行一次的任务 → 单次模式
  • 不确定用哪个 → 先选周期，后期再优化

8.5 避坑指南：我踩过的那些坑

做Gpt配置这么多年，我踩过的坑真不少。挑几个典型的跟大家说说：

  • 时钟源选错：有一次我选了外部晶振作为定时器时钟，结果晶振坏了，整个定时器都不跑了。后来我改成内部RC振荡器，虽然精度差一点，但至少不会完全停摆。
  • 中断优先级冲突：Gpt中断优先级设得太低，结果被其他中断一直抢占，定时器不准了。设得太高，又把关键任务打断了。这个平衡要把握好。
  • 回调函数重入：周期模式下，如果回调函数执行时间超过了定时周期，就会发生重入。上次回调还没执行完，下次触发又来了。这会导致数据混乱。解决办法是保证回调执行时间小于定时周期。

嗯，这些坑说起来都是泪。但踩过了，就记住了。你们现在知道这些，至少能少走一些弯路。

8.6 小结

Gpt模块虽然看起来简单，但它是整个系统的「节拍器」。配置得好，系统运行如丝般顺滑；配置不好，各种奇怪的问题都会冒出来。我建议大家在配置时，先想清楚需求，再动手。别一上来就开干，那样容易出问题。

下一章我们会讲Pwm模块，跟Gpt有很强的关联性。到时候我会讲讲怎么用Gpt来触发Pwm的更新，很有意思。今天就到这里，大家回去把EB Tresos打开，试着配一个10ms的周期定时器，感受一下。
