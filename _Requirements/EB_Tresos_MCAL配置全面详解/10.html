脉宽调制（Pwm）：Pwm模块概述、通道配置、占空比与频率设置、与Gpt的协同工作

各位工程师朋友，今天我们聊聊Pwm模块。说实话，Pwm在汽车电子里太常见了——从控制风扇转速、驱动电磁阀，到LED调光、蜂鸣器发声，几乎每个ECU都离不开它。我最早接触Pwm是在做车身域控制器的时候，当时为了调一个步进电机的转速，折腾了好几个晚上。嗯，今天就把这些经验分享给大家。

一、Pwm模块概述

Pwm，全称Pulse Width Modulation，脉宽调制。说白了，就是通过调节方波的高电平占整个周期的比例，来模拟一个可变的模拟电压。你想想看，一个信号要么高要么低，但通过调整占空比，就能让负载感受到不同的平均电压。

在AUTOSAR架构里，Pwm模块属于MCAL层，直接操作硬件定时器。它负责生成精确的PWM波形，支持多种通道配置。我个人习惯把Pwm模块看作一个“波形发生器”，你告诉它频率和占空比，它就老老实实输出信号。

核心概念：

  • 周期（Period）：一个完整方波的时间长度，单位通常是秒或微秒
  • 频率（Frequency）：周期的倒数，1秒内有多少个周期
  • 占空比（Duty Cycle）：高电平时间占整个周期的百分比
  • 极性（Polarity）：高电平有效还是低电平有效

我在项目中遇到过一个问题：同一个定时器模块被多个Pwm通道共用，结果频率设置冲突了。所以，理解Pwm的硬件资源分配非常重要。

二、通道配置

Pwm通道配置，是我们在EB Tresos里首先要做的事情。每个Pwm通道对应一个硬件输出引脚，你需要告诉工具：这个通道用哪个定时器、哪个比较通道、输出极性是什么。

配置项主要包括：

配置参数 说明 我的建议
PwmChannelId 通道ID，唯一标识 按功能命名，比如Pwm_FanCtrl
PwmChannelClass 通道类型：固定周期或可变周期 大部分场景用固定周期
PwmDutycycleDefault 默认占空比 设为0%，防止上电误动作
PwmPeriodDefault 默认周期 根据应用场景设定
PwmPolarity 输出极性：高有效或低有效 注意负载的驱动方式
PwmIdleState 空闲状态电平 通常与极性相反

举个例子，配置一个控制散热风扇的Pwm通道：

/* 在EB Tresos中的配置示例 */
PwmChannelConfigType {
    PwmChannelId = 0;
    PwmChannelClass = PWM_FIXED_PERIOD;
    PwmDutycycleDefault = 0;      /* 默认关闭 */
    PwmPeriodDefault = 10000;     /* 10ms周期，100Hz */
    PwmPolarity = PWM_HIGH_ACTIVE;
    PwmIdleState = PWM_LOW;
    PwmNotification = Pwm_Notification_Fan;  /* 回调函数 */
}

小技巧：我曾经在配置通道时，把PwmIdleState设成了高电平，结果ECU休眠时风扇还在转。后来我养成了一个习惯：空闲状态一定要和极性相反，这样在Pwm模块未初始化时，输出是安全的。

三、占空比与频率设置

占空比和频率，是Pwm的两个核心参数。频率决定了Pwm波形的快慢，占空比决定了有效输出的大小。

频率选择的原则：

  • 电机控制：通常20kHz以上，避免人耳听到啸叫
  • LED调光：100Hz-1kHz，太低会闪烁
  • 电磁阀驱动：几十Hz到几百Hz，看阀的响应时间
  • 加热器：几Hz就够了，热惯性大

我记得有一次做LED尾灯项目，用了200Hz的Pwm，结果在摄像头下看到明显的闪烁。后来改到1kHz，问题解决了。所以，频率选择要考虑整个系统的感知范围。

占空比设置：

在EB Tresos里，占空比通常用百分比或定时器计数值表示。我个人更喜欢用计数值，因为更精确。比如一个10位定时器，周期是1024，那么50%占空比就是512。

/* 动态设置占空比的API */
void Pwm_SetDutyCycle(uint8 ChannelId, uint16 DutyCycleValue);

/* 示例：设置通道0占空比为75% */
Pwm_SetDutyCycle(0, 768);  /* 假设周期为1024 */

注意：占空比不能超过周期值。我曾经在代码里写了个bug，占空比设得比周期还大，结果Pwm输出直接变成了常高。调试了半天才发现是参数越界了。所以，调用Pwm_SetDutyCycle之前，一定要做范围检查。

四、与Gpt的协同工作

Pwm和Gpt（通用定时器）的关系，就像演员和导演。Gpt负责提供时间基准，Pwm负责输出波形。在AUTOSAR里，Pwm模块通常依赖Gpt模块来获取时钟源。

协同工作的典型场景：

  • Pwm使用Gpt的某个定时器作为时基
  • Gpt产生周期中断，Pwm在中断里更新占空比
  • 多个Pwm通道共享一个Gpt定时器

配置时，你需要在EB Tresos里指定Pwm通道关联的Gpt设备。比如：

/* Pwm与Gpt的关联配置 */
PwmConfigType {
    PwmGptDeviceRef = Gpt_Device_0;  /* 使用Gpt设备0 */
    PwmGptChannelRef = Gpt_Channel_2; /* 使用Gpt通道2 */
    PwmPrescaler = 64;                /* 预分频 */
}

为什么会这样设计？因为硬件定时器资源有限，通过Gpt统一管理，可以避免资源冲突。我见过一个项目，工程师直接在Pwm里硬编码了定时器地址，结果换了芯片平台后，所有代码都要重写。用Gpt做抽象层，移植性就好很多。

协同工作的注意事项：

  1. 确保Gpt的时钟频率和Pwm的期望频率匹配
  2. Gpt中断优先级要合理设置，避免Pwm更新被延迟
  3. 多个Pwm通道共享Gpt时，注意通道间的相互影响
  4. 休眠模式下，Gpt和Pwm的时钟管理要一致

嗯，这里还要提一个坑。我曾经在调试时发现，Pwm输出偶尔会跳变。查了很久才发现，是Gpt中断里更新占空比的代码，和Pwm硬件寄存器更新产生了竞争。解决方案是在Gpt中断里加一个标志位，等Pwm当前周期结束后再更新。

/* 安全的Pwm更新方式 */
void Gpt_Notification(void)
{
    /* 设置更新标志 */
    PwmUpdateFlag = TRUE;
}

void main_loop(void)
{
    if (PwmUpdateFlag)
    {
        /* 在Pwm周期边界更新 */
        Pwm_SetDutyCycle(0, newDuty);
        PwmUpdateFlag = FALSE;
    }
}

避坑指南：我曾经在Gpt中断里直接调用Pwm_SetDutyCycle，结果导致Pwm波形毛刺。后来我改成在中断里只设标志，在主循环里更新，波形就干净了。记住：中断服务函数要短小精悍，别在里面干重活。

总结一下今天的要点：Pwm模块配置要关注通道参数、频率和占空比的合理选择，以及和Gpt的协同工作。这些内容在EB Tresos里都有对应的配置界面，关键是要理解背后的硬件原理。下次我们讲Icu模块，也就是输入捕获，到时候再聊。
