看门狗（Wdg）：Wdg模块概述、内部与外部看门狗配置、触发模式、安全机制与故障处理

看门狗这个模块，说实话，在嵌入式系统里就像个「老黄牛」——平时不声不响，一旦系统跑飞了，它才是最后的救命稻草。我做了这么多年汽车电子，见过不少因为看门狗配置不当导致的现场事故，有的车在高速上直接黑屏重启，想想都后怕。

今天咱们就好好聊聊EB Tresos里的Wdg模块。我会把内部看门狗、外部看门狗、触发模式这些核心内容掰开揉碎了讲，顺便分享一些我踩过的坑。

一、Wdg模块到底在干什么？

看门狗的本质，就是一个计数器。它不停地从某个最大值往下减，减到0就触发复位。正常工作时，软件得在它减到0之前「喂狗」——也就是把计数器重新置回最大值。

如果软件卡死了，没人喂狗，计数器就会一直减到0，然后系统复位。说白了，这就是个「你不干活我就重启你」的机制。

在AUTOSAR架构里，Wdg模块负责管理这个机制。它不直接操作硬件寄存器，而是通过MCAL层的驱动去控制具体的看门狗硬件。我个人习惯把Wdg模块分成三层来看：

  • 应用层：调用Wdg_Trigger()喂狗
  • Wdg模块：管理触发模式、安全策略
  • MCAL驱动：直接操作寄存器

嗯，这里要注意，很多新手容易把Wdg模块和WdgIf模块搞混。WdgIf是接口层，负责把多个看门狗实例统一管理，而Wdg模块是具体的驱动实现。咱们今天主要讲的是MCAL层的Wdg驱动。

二、内部看门狗 vs 外部看门狗

看门狗分两种：内部的和外部的。这个选择在项目初期就得定下来，不然后期改起来非常痛苦。

1. 内部看门狗

内部看门狗就是芯片自带的，比如Infineon TC3xx的WDT、NXP S32K的WDOG。它的优点是成本低、不需要额外硬件、配置方便。但缺点也很明显——如果芯片本身挂了（比如时钟停振），内部看门狗也跟着完蛋。

我在一个项目里就遇到过这种情况。当时用的是内部看门狗，结果芯片因为电源纹波太大导致内核跑飞，内部看门狗也跟着失效了。从那以后，我对内部看门狗就多留了个心眼。

在EB Tresos里配置内部看门狗，主要关注这几个参数：

参数 说明 我的建议
WdgTimeout 超时时间，单位ms 一般设100-500ms，别太短也别太长
WdgTriggerMode 触发模式 后面会细讲
WdgPrescaler 预分频系数 根据时钟频率算一下，别超范围
WdgWindowMode 窗口模式使能 安全等级高的项目建议开启

2. 外部看门狗

外部看门狗是独立的芯片，比如SP706、MAX706、或者专用的汽车级看门狗如TPS3431。它通过一个GPIO引脚和主芯片通信，主芯片定期给这个引脚一个脉冲（喂狗），如果长时间没脉冲，外部看门狗就拉低复位引脚。

外部看门狗的好处是：即使主芯片彻底挂了（包括时钟、电源），它依然能工作。所以ASIL-D级别的项目，我强烈建议用外部看门狗。

配置外部看门狗时，EB Tresos里需要设置：

  • 喂狗引脚：哪个GPIO用来输出脉冲
  • 脉冲宽度：一般几微秒到几十微秒
  • 超时时间：由外部芯片决定，软件里要匹配
  • 初始状态：上电后看门狗是否立即生效
⚠️ 重要提醒： 外部看门狗的喂狗引脚千万别复用！我曾经见过一个项目，工程师把喂狗引脚同时用作普通IO，结果调试时不小心拉低了，整车直接复位。这种低级错误，犯一次就够你记一辈子。

三、触发模式：你到底怎么喂狗？

触发模式，说白了就是「软件怎么告诉看门狗我还活着」。AUTOSAR里定义了三种模式：

1. 直接触发模式

最简单粗暴的方式。软件调用Wdg_Trigger()，驱动直接写寄存器喂狗。适合简单的应用，但有个问题——如果喂狗代码本身跑飞了，那就完蛋了。

代码示例：

/* 直接触发模式 */
void MyTask_1ms(void)
{
    /* 做一些工作 */
    ...
    
    /* 喂狗 */
    Wdg_Trigger(0);  /* 0表示看门狗实例0 */
}

2. 窗口触发模式

这个就高级一点了。它规定了一个时间窗口：太早喂狗不行（说明程序跑太快，可能跳过了某些检查），太晚喂狗也不行（说明程序卡住了）。只有在窗口内喂狗才算有效。

窗口模式有两个关键参数：

  • WdgWindowStart：窗口起始时间，比如超时时间的30%
  • WdgWindowEnd：窗口结束时间，比如超时时间的70%

举个例子，如果超时时间是100ms，窗口从30ms到70ms。那么你只能在30ms到70ms之间喂狗，早了晚了都会触发复位。

💡 我的经验： 窗口模式特别适合安全等级高的项目。我曾经在一个ADAS项目里用了窗口模式，配合任务执行时间监控，成功捕获了好几次任务调度异常。如果你做的是ASIL-B以上的项目，建议开启窗口模式。

3. 偏移触发模式

这个模式比较少见，它允许你设置多个喂狗点，每个点有不同的偏移时间。适合多任务场景，每个任务在各自的时间点喂狗。

配置示例：

/* 偏移触发模式配置 */
Wdg_ConfigType WdgConfig = {
    .TriggerMode = WDGMODE_OFFSET,
    .OffsetPoints = 3,
    .OffsetTime[0] = 20,   /* 任务1在20ms喂狗 */
    .OffsetTime[1] = 50,   /* 任务2在50ms喂狗 */
    .OffsetTime[2] = 80    /* 任务3在80ms喂狗 */
};

四、安全机制与故障处理

看门狗本身也得有安全机制，不然它自己出问题了怎么办？AUTOSAR的Wdg模块提供了几种故障处理方式：

1. 故障检测

Wdg模块会检测以下几种故障：

  • 超时故障：计数器减到0了
  • 窗口违规：在窗口外喂狗
  • 配置错误：参数配置不合理
  • 硬件故障：看门狗硬件本身出问题

2. 故障响应

检测到故障后，Wdg模块可以执行以下操作：

响应方式 说明 适用场景
WdgReset 直接复位系统 严重故障，比如超时
WdgInterrupt 触发中断，让软件处理 窗口违规，可以记录日志
WdgIgnore 忽略故障，继续运行 调试阶段，别用在量产车上
🔑 关键点： 我个人建议，量产项目里超时故障一定要触发复位，别想着用中断去处理。为什么？因为系统已经卡死了，中断服务程序可能也跑不了。直接复位是最可靠的。

3. 故障恢复

复位之后怎么办？Wdg模块支持故障恢复机制：

  • 冷启动：完全复位，所有状态清零
  • 热启动：保留部分状态，快速恢复

我建议在故障恢复时，记录一下复位原因。比如在NVM里存一个标志位，下次启动时读取，就知道是看门狗复位还是上电复位。这个信息对排查问题非常有帮助。

五、避坑指南

最后，分享几个我踩过的坑，希望能帮你少走弯路：

  1. 喂狗位置别乱放：我曾经把喂狗放在中断里，结果主循环卡死了，中断还在跑，看门狗永远不超时。系统死了都没人知道。
  2. 调试时记得关看门狗：调试阶段，看门狗会频繁复位，影响调试效率。EB Tresos里有个WdgDevErrorDetect开关，调试时打开，量产时关掉。
  3. 多核芯片要注意：有些多核芯片，每个核有自己的看门狗。配置时要搞清楚哪个核负责喂哪个狗，别喂错了。
  4. 休眠模式要处理：系统进入休眠时，看门狗要不要继续跑？这个得根据项目需求来。一般建议休眠时暂停看门狗，但唤醒后要重新初始化。
