第22章 CRC校验（Crc）：Crc模块功能、多项式选择、计算模式与硬件加速

各位同学，今天我们来聊聊CRC校验。说实话，CRC在汽车嵌入式里是个「小透明」——平时没人注意它，但一旦通信数据出错，它就成了救命稻草。我在做第一个基于CAN的项目时，就吃过没好好配CRC的亏，后来才明白这玩意儿有多重要。

22.1 Crc模块到底在干什么？

简单说，CRC模块就是给数据算个「指纹」。发送方算一遍，接收方也算一遍，两个指纹对得上，数据就没问题。对不上？那肯定是在传输过程中被篡改了。

在AUTOSAR架构里，Crc模块属于MCAL层，直接跟硬件打交道。它提供了一套标准接口，上层模块（比如Can、Lin、Ethernet）都能调用它来做校验。我个人习惯是把Crc模块当作一个「公共服务」，所有通信协议栈都共用它。

它的核心功能就三个：

  • 计算校验值：给一段数据算CRC
  • 验证数据完整性：对比两个CRC值是否一致
  • 支持多种多项式：不同协议用不同的多项式，Crc模块得都支持

你想想看，如果每个通信模块都自己写一套CRC算法，那代码得多乱？AUTOSAR把这部分统一了，省心不少。

22.2 多项式选择——不是随便挑一个就行

多项式是CRC的核心。不同的多项式，检错能力、计算速度都不一样。我在项目中遇到过有人随便选了个多项式，结果数据错了好几个字节都没检出来……嗯，那场面挺尴尬的。

常见的多项式有这么几种：

名称 多项式（十六进制） 典型应用
CRC-8 0x07 SMBus、1-Wire
CRC-16-IBM 0x8005 Modbus、USB
CRC-16-CCITT 0x1021 X.25、HDLC
CRC-32 0x04C11DB7 Ethernet、ZIP

选多项式时，我建议你遵循两个原则：

  • 看协议要求：CAN协议通常用CRC-15（多项式0x4599），Ethernet用CRC-32。别自己发明，用标准的多项式。
  • 看数据长度：数据短（比如几个字节），CRC-8就够了。数据长（几百字节），至少得上CRC-16。
注意： 多项式选错了，CRC计算出来的值跟协议不匹配，通信双方对不上，整个系统就崩了。我曾经见过一个团队，因为多项式高低位搞反了，排查了两天才找到问题。

22.3 计算模式——8位、16位还是32位？

CRC的计算模式，说白了就是一次算多少位。AUTOSAR的Crc模块支持三种模式：

  • CRC-8：一次算8位，结果也是8位。适合小数据量，比如传感器数据。
  • CRC-16：一次算16位，结果16位。用得最多，CAN、LIN都用它。
  • CRC-32：一次算32位，结果32位。大数据量、高可靠性场景，比如Ethernet。

我个人的经验是：别盲目追求高位。CRC-32虽然检错能力强，但计算开销也大。如果你的数据只有几十个字节，CRC-16完全够用。你想想看，为了多检错那0.0001%的概率，让CPU多跑几十微秒，值不值？

在EB Tresos里配置时，你需要指定：

  • CrcMode：选8、16还是32
  • CrcPolynomial：多项式值
  • CrcInitialValue：初始值，通常是0xFFFF或0x0000
  • CrcXorValue：异或值，计算完后再异或一下

举个例子，配置CRC-16-CCITT：

CrcMode = CRC_16
CrcPolynomial = 0x1021
CrcInitialValue = 0xFFFF
CrcXorValue = 0x0000
CrcReflectIn = TRUE
CrcReflectOut = TRUE

这里有个坑：CrcReflectIn和CrcReflectOut。这两个参数控制输入数据和输出结果是否要「反转」位序。很多协议要求反转，比如Modbus。忘了配这个，算出来的CRC值就是错的。

22.4 硬件加速与软件回退——性能与可靠性的平衡

现代MCU很多都内置了CRC硬件模块。硬件算CRC有多快？我测过，同样的数据，硬件比软件快几十倍。尤其是大数据量，硬件几乎不占CPU时间。

在EB Tresos里，你可以配置：

  • CrcHwUnit：选择哪个硬件单元（如果有多个）
  • CrcHwMode：硬件模式，比如自动计算还是手动触发
  • CrcSwFallback：软件回退使能

硬件加速的好处很明显：

  • 速度快，不占CPU
  • 功耗低，硬件算完就完事
  • 结果稳定，不受软件干扰

但硬件也有缺点：

  • 灵活性差，只能算固定的多项式
  • 如果硬件坏了，CRC就废了

所以AUTOSAR引入了软件回退机制。说白了就是：硬件能用就用硬件，硬件出问题了，自动切到软件算法。我在一个项目中就遇到过硬件CRC模块被意外关闭的情况，幸好配了软件回退，系统没崩。

我的建议： 不管硬件多可靠，都打开软件回退。这就像开车系安全带——你可能一辈子用不上，但用上的时候能救命。配置时把CrcSwFallback设为TRUE，再给软件算法配好多项式就行。

22.5 避坑指南——我踩过的那些坑

做CRC配置这么多年，我总结了几条血泪教训：

  1. 多项式别搞反：有些文档给的多项式是「正常」的，有些是「反转」的。配之前先确认协议文档的写法。
  2. 初始值别乱设：0xFFFF和0x0000算出来的结果完全不同。查协议，别猜。
  3. 别忘了异或值：很多协议最后一步要异或一个值，比如0xFFFF。忘了这步，CRC值对不上。
  4. 测试要覆盖边界：空数据、全0数据、全1数据，都得测一遍。我曾经遇到过空数据时硬件CRC返回0，但软件算法返回非0的情况。

嗯，说到测试，我建议你在集成测试阶段专门写个CRC测试用例。把已知数据丢进去，看算出来的值跟预期是否一致。这一步花不了多少时间，但能省掉后面排查的大把功夫。

22.6 小结

CRC校验看着简单，但细节不少。多项式选对、模式配准、硬件加速打开、软件回退备好——这几步走稳了，CRC模块基本不会出问题。记住一句话：CRC不是万能的，但没有CRC是万万不能的。
