第十七章 直接内存访问（DMA）：解放CPU的数据搬运工

各位同学，今天我们来聊聊DMA。说实话，在嵌入式系统里，DMA是个特别有意思的模块。它就像个勤快的搬运工，专门负责在内存和外设之间搬数据，而且不需要CPU操心。我刚开始接触AUTOSAR时，总觉得DMA可有可无，直到有一次项目里CPU负载飙到90%以上，我才意识到——没有DMA，CPU就是个被数据搬运累垮的苦力。

17.1 DMA模块原理：为什么需要它？

DMA的全称是Direct Memory Access，直接内存访问。它的核心思想很简单：让硬件自己搬数据，CPU只管算。

你想想看，传统方式下，CPU要从外设读数据，得先轮询或等中断，然后一条条指令把数据从外设寄存器搬到内存。这个过程CPU全程参与，但干的活其实很机械——就是复制粘贴。DMA就是专门干这个的硬件模块，它有自己的控制器，能直接访问系统总线，把数据从一个地址搬到另一个地址。

我在项目里遇到过这样的情况：一个高速ADC每秒产生几兆字节的数据，如果用CPU去搬，那CPU基本就废了，啥也别干了。用了DMA之后，CPU占用率直接降到5%以下。说白了，DMA就是给CPU减负的。

DMA的核心价值：

  • 解放CPU：让CPU从数据搬运中解脱出来
  • 提高吞吐量：硬件级的数据传输，速度远快于软件
  • 降低延迟：响应速度快，适合实时性要求高的场景
  • 支持后台传输：CPU可以同时处理其他任务

17.2 DMA通道配置：每个通道都是一个独立搬运工

DMA模块通常包含多个通道，每个通道可以独立配置。在EB Tresos里配置DMA时，有几个关键参数必须搞清楚。

参数 说明 我的建议
源地址 数据从哪里来（外设寄存器或内存地址） 注意地址对齐，有些MCU要求32位对齐
目标地址 数据搬到哪里去 确保目标缓冲区足够大，别溢出
传输宽度 每次传输多少位（8/16/32位） 匹配外设数据宽度，否则会丢数据
传输计数 总共要传输多少次 循环模式下这个值要小心设置
地址增量 每次传输后地址是否自动增加 外设到内存通常源地址不变，目标地址递增

嗯，这里要注意一个坑：地址增量模式。我记得有一次配置SPI的DMA传输，源地址设成了递增模式，结果每次读到的数据都错位了。后来才发现，外设的数据寄存器地址是固定的，不应该递增。你想想看，外设寄存器就一个地址，你递增了不就读到别的地方去了吗？

17.3 传输模式：单次、循环、链表

DMA的传输模式有三种，每种都有它的用武之地。我个人习惯根据应用场景来选择，而不是一味追求复杂。

17.3.1 单次传输模式

这是最基础的模式。配置好源地址、目标地址和传输计数后，启动一次DMA，搬完指定数量的数据就停止。适合一次性数据传输，比如把配置数据从内存搬到外设。

/* 单次传输配置示例 */
Dma_ConfigType DmaConfig = {
    .DmaChannel = 0,
    .SrcAddr = (uint32)&SPI_TX_REG,
    .DestAddr = (uint32)txBuffer,
    .TransferCount = 256,
    .SrcAddrMode = DMA_ADDR_FIXED,    /* 外设地址固定 */
    .DestAddrMode = DMA_ADDR_INC,     /* 内存地址递增 */
    .TransferMode = DMA_MODE_SINGLE
};

17.3.2 循环传输模式

这个模式很实用。传输完成后自动重新开始，适合持续的数据流。比如音频播放、ADC连续采样等场景。

我的经验：循环模式配合双缓冲（ping-pong buffer）效果最好。一个缓冲区在传输，另一个缓冲区让CPU处理数据，交替使用，效率极高。我曾经在一个音频项目里用这个方案，CPU几乎零开销就完成了音频数据的搬移。

17.3.3 链表传输模式

这是最灵活的模式，也是AUTOSAR里比较高级的用法。你可以预先定义多个传输描述符（Descriptor），每个描述符指定一次传输的配置。DMA会自动按链表顺序执行这些传输。

/* 链表传输描述符示例 */
Dma_Descriptor_t desc1 = {
    .SrcAddr = (uint32)buffer1,
    .DestAddr = (uint32)peripheral1,
    .Count = 128,
    .NextDesc = &desc2    /* 指向下一个描述符 */
};

Dma_Descriptor_t desc2 = {
    .SrcAddr = (uint32)buffer2,
    .DestAddr = (uint32)peripheral2,
    .Count = 64,
    .NextDesc = NULL      /* 链表结束 */
};

说白了，链表模式就是让DMA自己知道下一步该干什么，不需要CPU一次次重新配置。这在处理复杂的数据流时特别有用，比如图像处理中的多行数据传输。

17.4 与外设的联动：DMA不是孤岛

DMA要发挥作用，必须和外设紧密配合。在AUTOSAR架构里，DMA通常和SPI、UART、ADC、定时器等外设协同工作。

联动机制的核心是硬件触发。外设产生一个请求信号，DMA收到后就开始传输。比如SPI发送完成时产生一个DMA请求，DMA就把下一个数据从内存搬到SPI的发送寄存器。

我曾经踩过的坑：外设的DMA请求优先级和DMA通道的优先级要匹配。有一次我把一个高频率的外设配成了低优先级DMA通道，结果数据频繁丢失。后来把优先级调高，问题就解决了。记住：高频外设配高优先级通道，低频外设配低优先级通道。

在EB Tresos里配置外设联动时，有几个关键点：

  • 触发源选择：明确是哪个外设的哪个事件触发DMA
  • 触发极性：上升沿触发还是电平触发，这个要看外设手册
  • 传输完成中断：是否在传输完成后产生中断通知CPU
  • 错误处理：传输错误时DMA的行为，是重试还是停止

举个例子，配置SPI的DMA接收：

/* SPI DMA接收联动配置 */
/* 1. 配置SPI：使能DMA接收请求 */
SPI_CTRL_REG |= SPI_DMA_RX_EN;

/* 2. 配置DMA通道：源地址为SPI接收寄存器，目标地址为内存缓冲区 */
DMA_CH_CTRL.SRC_ADDR = (uint32)&SPI_RX_REG;
DMA_CH_CTRL.DST_ADDR = (uint32)rxBuffer;
DMA_CH_CTRL.TRANSFER_COUNT = 1024;
DMA_CH_CTRL.TRIGGER_SOURCE = DMA_TRIG_SPI_RX;
DMA_CH_CTRL.TRIGGER_EDGE = DMA_EDGE_RISING;

/* 3. 启动DMA通道 */
DMA_CH_CTRL.ENABLE = 1;

你可能会问：为什么不用中断？中断也可以啊。但你想过没有，如果每接收一个字节就产生一次中断，那CPU得频繁进出中断服务程序，开销很大。DMA可以攒够一批数据再通知CPU，效率高得多。

17.5 实际项目中的DMA配置策略

根据我的项目经验，DMA配置有几个原则：

  1. 能不配就不配：不是所有外设都需要DMA，低频外设用中断就够了
  2. 通道复用要谨慎：多个外设共享一个DMA通道时，要做好仲裁
  3. 缓冲区管理是关键：DMA是硬件在搬数据，CPU和DMA不能同时访问同一块内存
  4. 调试时先关DMA：我习惯先不用DMA调通基本功能，再加DMA优化

避坑指南：我曾经在一个项目里，DMA配置好后死活不工作。查了两天，最后发现是时钟没使能。DMA模块的时钟在系统启动时默认是关闭的，必须在RCC配置里打开。这种低级错误，说多了都是泪。

好了，关于DMA的原理和配置，今天就讲到这里。下一章我们会聊PWM模块，那个也很有意思。记住，DMA是嵌入式系统里提升性能的利器，但用不好也会带来麻烦。多实践，多踩坑，慢慢就熟练了。
