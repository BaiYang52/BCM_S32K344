第16章 以太网（Eth）：Eth模块概述、MAC与PHY配置、DMA描述符链、中断与轮询模式

以太网模块，在AUTOSAR架构里算是个「大块头」。为什么这么说？因为它既要处理高速数据流，又要兼顾实时性。我最早接触Eth模块时，是在一个车载摄像头项目上，那时候对DMA描述符链的理解还不够深，结果数据丢包丢得我头皮发麻。今天咱们就把这块硬骨头啃下来。

16.1 Eth模块概述

Eth模块，说白了就是MCU和外部以太网物理层之间的「翻译官」。它负责把上层协议栈的数据包，转换成能在网线上传输的电信号。反过来也一样。

在AUTOSAR分层里，Eth模块属于MCAL层。它直接操作硬件寄存器，不依赖任何操作系统。嗯，这里要注意：Eth模块的配置，直接影响整个车载网络的通信质量。

我个人的习惯是，在开始配置之前，先搞清楚三个关键角色：

  • MAC层：媒体访问控制，负责数据帧的封装和解封装
  • PHY层：物理层，负责信号的编码解码和传输
  • MII/RMII接口：MAC和PHY之间的桥梁

你想想看，如果MAC和PHY的配置不匹配，那数据根本传不出去。我曾经在一个项目里，就因为MII接口的时钟极性配反了，整整排查了两天。

16.2 MAC与PHY配置

MAC配置，核心是设置工作模式。EB Tresos里，你需要关注这几个参数：

参数 说明 我的建议
EthMacController MAC控制器实例 一般选Eth_0
EthMacDuplex 全双工/半双工 车载场景建议全双工
EthMacSpeed 10M/100M/1000M 根据PHY能力选
EthMacAddress MAC地址 注意唯一性

PHY配置相对复杂一些。因为不同厂家的PHY芯片，寄存器地址和功能都不一样。EB Tresos提供了通用的PHY抽象层，但底层驱动还是要自己适配。

我记得有一次，项目里用了博通的PHY，结果EB Tresos默认的PHY驱动不兼容。最后我不得不手动修改PHY的初始化序列，把自动协商关掉，强制设置成100M全双工。嗯，这种坑，踩过一次就记住了。

关键点：MAC和PHY的配置，一定要保证速度、双工模式、接口类型完全一致。否则链路起不来。

16.3 DMA描述符链

DMA描述符链，是Eth模块性能的关键。说白了，它就是一个环形缓冲区，用来管理数据包的收发。

每个描述符包含：

  • 数据缓冲区的地址
  • 数据长度
  • 控制标志（如：是否最后一个描述符）
  • 状态信息（如：传输完成标志）

在EB Tresos里，你需要配置描述符的数量和大小。我个人习惯，接收描述符至少配16个，发送描述符8个。为什么？因为接收方向的数据流通常更大，描述符少了容易丢包。

来看一个典型的描述符链配置代码：

/* 描述符链配置示例 */
#define ETH_RX_DESC_NUM    16
#define ETH_TX_DESC_NUM    8

/* 接收描述符数组 */
Eth_DescType RxDesc[ETH_RX_DESC_NUM];
/* 发送描述符数组 */
Eth_DescType TxDesc[ETH_TX_DESC_NUM];

/* 初始化描述符链 */
void Eth_InitDescChain(void)
{
    uint32 i;
    
    for(i = 0; i < ETH_RX_DESC_NUM; i++)
    {
        RxDesc[i].BufferAddr = (uint32)RxBuffer[i];
        RxDesc[i].Length = ETH_BUFFER_SIZE;
        RxDesc[i].Control = ETH_DESC_OWN_BIT;  /* 交给DMA控制 */
        RxDesc[i].NextDesc = &RxDesc[(i+1) % ETH_RX_DESC_NUM];
    }
    
    /* 发送描述符类似，但初始时归CPU控制 */
    for(i = 0; i < ETH_TX_DESC_NUM; i++)
    {
        TxDesc[i].Control = 0;  /* CPU控制 */
    }
}

这里有个容易踩的坑：描述符的NextDesc指针，一定要形成闭环。我曾经见过一个同事，把最后一个描述符的NextDesc指向了NULL，结果DMA跑到最后一个描述符就停了，再也收不到新数据。

警告：描述符链的缓冲区地址，必须是物理地址。如果你用了MMU，记得做地址转换。否则DMA会访问到错误的内存区域。

16.4 中断与轮询模式

Eth模块支持两种数据收发模式：中断模式和轮询模式。怎么选？看应用场景。

中断模式：

  • 适合实时性要求高的场景
  • 每个数据包到达都会触发中断
  • CPU利用率低，但中断频繁
  • 配置时要注意中断优先级

轮询模式：

  • 适合大数据量、低延迟要求的场景
  • CPU不断查询描述符状态
  • 没有中断开销，但CPU占用高
  • 常用于诊断或刷写场景

我个人的经验是：正常通信用中断模式，刷写或大数据传输时切换到轮询模式。为什么？因为刷写时数据量巨大，中断频繁会导致CPU频繁进出中断上下文，反而降低效率。

来看中断模式的处理流程：

/* 中断服务函数 */
void Eth_IrqHandler(void)
{
    uint32 status;
    
    status = Eth_GetInterruptStatus();
    
    if(status & ETH_INT_RX)
    {
        /* 处理接收数据 */
        Eth_ProcessRxDescriptor();
    }
    
    if(status & ETH_INT_TX)
    {
        /* 处理发送完成 */
        Eth_ProcessTxDescriptor();
    }
    
    /* 清除中断标志 */
    Eth_ClearInterrupt(status);
}

轮询模式就简单多了：

/* 轮询模式主循环 */
void Eth_PollingLoop(void)
{
    while(1)
    {
        /* 检查接收描述符 */
        if(RxDesc[RxIndex].Control & ETH_DESC_OWN_BIT == 0)
        {
            /* 处理接收到的数据包 */
            ProcessRxPacket(&RxDesc[RxIndex]);
            /* 重新初始化描述符 */
            RxDesc[RxIndex].Control = ETH_DESC_OWN_BIT;
            RxIndex = (RxIndex + 1) % ETH_RX_DESC_NUM;
        }
        
        /* 检查发送描述符 */
        if(TxDesc[TxIndex].Control & ETH_DESC_OWN_BIT == 0)
        {
            /* 发送完成，可以发送下一个包 */
            TxIndex = (TxIndex + 1) % ETH_TX_DESC_NUM;
        }
    }
}

小技巧：在EB Tresos里，你可以同时使能中断模式和轮询模式。平时用中断，需要刷写时通过Eth_SetControllerMode切换到轮询。这样既保证了实时性，又兼顾了大数据传输的效率。

最后说一句：Eth模块的调试，最好用逻辑分析仪抓MII接口的波形。光靠软件调试，很多问题根本看不出来。我曾经有一个PHY链路不稳定的问题，软件查了三天没结果，最后用逻辑分析仪一看，原来是PHY的复位时序不对。嗯，硬件问题，软件背锅，这种事在嵌入式开发里太常见了。
