第十三章：内部集成电路（I2c）——I2c模块功能、总线时序配置、主从模式、多主机仲裁处理

好，咱们今天聊聊I2C。说实话，这个协议在汽车电子里太常见了。传感器、执行器、甚至一些诊断芯片，都靠它来通信。我刚开始接触AUTOSAR MCAL时，觉得I2C配置挺简单的，不就是两根线嘛。结果第一次调板子，数据死活读不对，波形一塌糊涂。嗯，从那以后，我再也不敢小看I2C的时序配置了。

13.1 I2c模块的核心功能

I2C模块在EB Tresos里，说白了就是管理那两根线——SCL（时钟）和SDA（数据）。它的核心任务有三个：

  • 产生总线时序：按照你设定的速率，把时钟和数据送出去。
  • 处理地址匹配：作为从机时，能识别自己的地址；作为主机时，能发起通信。
  • 仲裁与错误处理：多主机抢总线时，能优雅地退让或重试。

我个人习惯，在配置之前，先想清楚这个I2C模块是当主机用，还是当从机用。因为AUTOSAR的I2c模块，主从模式是分开配置的，不能混着来。

关键点：I2C模块在MCAL里通常只支持一种角色。要么是主机，要么是从机。别想着一个模块同时干两件事，那得用两个实例。

13.2 总线时序配置——别让时钟毁了你的通信

时序配置，是I2C最容易出问题的地方。我见过太多人，直接把时钟频率设成400kHz，结果上板子就挂。为什么？因为总线上的上拉电阻、电容负载，都会影响实际波形。

在EB Tresos里，时序配置主要看这几个参数：

参数名 说明 我的建议
I2cClockFrequency 模块输入时钟频率（通常是PCLK） 先确认你的MCU时钟树，别想当然
I2cBaudRate 目标波特率（100kHz/400kHz/1MHz） 标准模式100k，快速模式400k，高速模式1M
I2cRiseTime 上升时间（ns） 这个值跟PCB走线有关，我一般先设100ns
I2cFallTime 下降时间（ns） 通常比上升时间短，设30-50ns
I2cSdaHoldTime SDA保持时间 这个值太小，从机可能采样不到

举个例子，你配置400kHz快速模式，输入时钟40MHz。工具会自动计算分频系数和占空比。但如果你不设上升时间，它可能按理想情况算，实际波形就变形了。

我的经验：我曾经在一个项目里，I2C通信偶尔丢数据。查了半天，发现是SDA保持时间设得太短，从机来不及采样。把保持时间从0.1us改成0.3us，问题就解决了。所以，别太相信默认值。

13.3 主从模式配置——角色不同，配置天差地别

主模式和从模式，在EB Tresos里是两个不同的配置路径。我简单说说区别。

13.3.1 主机模式配置

主机模式，你得告诉模块：

  • 什么时候发起START信号
  • 发送什么地址（7位还是10位）
  • 读写方向
  • 什么时候发STOP信号

在代码里，你通常要调用I2c_MasterTransmit()或I2c_MasterReceive()。这些API会帮你封装好时序。但底层配置，你得把中断或DMA安排好。我个人习惯用中断，因为DMA虽然快，但处理异常情况时比较麻烦。

/* 主机发送示例（伪代码） */
I2c_ConfigType config;
config.masterMode = TRUE;
config.baudRate = 400000;  /* 400kHz */
config.address = 0x50;     /* 从机地址 */
config.direction = I2C_WRITE;

I2c_Init(&config);
I2c_MasterTransmit(0x50, dataBuffer, length);

13.3.2 从机模式配置

从机模式，你得告诉模块：

  • 自己的地址（可以配多个地址，比如7位+10位）
  • 收到地址匹配后，怎么响应
  • 数据接收/发送的回调函数

从机模式有个坑：地址匹配后，你得及时响应。如果CPU太忙，没来得及读数据，总线就可能被卡住。我曾经遇到过，从机地址匹配后，主机一直等ACK，结果从机没反应，总线就锁死了。解决办法是，把I2C中断优先级设高一点。

注意：从机模式下，如果使了时钟拉伸（Clock Stretching），一定要确认从机能不能拉低SCL。有些便宜芯片不支持这个功能，配置了反而出问题。

13.4 多主机仲裁处理——抢总线，得有规矩

多主机仲裁，是I2C协议里比较高级的功能。说白了，就是两个主机同时想发数据，谁先抢到总线？

仲裁的规则很简单：谁先拉低SDA，谁就赢。输的那个，得乖乖退让，等下次机会。

在EB Tresos里，多主机配置主要关注这几个点：

  • 仲裁丢失中断：一定要使能。当你的模块检测到仲裁丢失时，会触发中断。你得在中断服务函数里，停止当前传输，重新尝试。
  • 总线空闲检测：多主机环境下，每个主机都要能检测总线是否空闲。通常用STOP信号来判断。
  • 重试机制：仲裁丢失后，不能马上重试。我一般会加一个随机延时，避免两个主机同时重试，又撞车。

避坑指南：我曾经在一个项目里，两个ECU通过I2C通信，都配成了主机。结果总线经常锁死。查了三天，发现是仲裁丢失中断没使能。一个主机丢了仲裁，还在那傻等ACK，另一个主机也等回复，两个就死锁了。使能中断后，加上重试逻辑，问题解决。

多主机仲裁的代码逻辑，大致是这样的：

/* 仲裁丢失中断处理 */
void I2c_ArbitrationLostHandler(void)
{
    /* 停止当前传输 */
    I2c_StopTransfer();
    
    /* 随机延时，避免再次冲突 */
    delay_ms(random(1, 10));
    
    /* 重新尝试发送 */
    I2c_MasterTransmit(slaveAddr, data, len);
}

你想想看，如果没有这个随机延时，两个主机同时重试，是不是又得打架？所以，多主机环境，软件设计比硬件配置更重要。

13.5 总结一下

I2C模块配置，说难不难，说简单也不简单。核心就三点：

  1. 时序要算准：上升时间、下降时间、保持时间，别用默认值。
  2. 主从要分清：角色不同，配置路径和API都不同。
  3. 仲裁要处理：多主机环境下，中断和重试机制必须到位。
