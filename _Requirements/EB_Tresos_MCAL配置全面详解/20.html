第二十讲：EEPROM模拟（Eep）—— 从原理到实战

各位同学，今天我们来聊聊EEPROM模拟，也就是Eep模块。说实话，这个模块在MCAL里属于那种「平时不起眼，出事真要命」的角色。我最早接触它是在一个车身控制器的项目上，当时为了省成本，MCU没带真正的EEPROM，只能用Flash模拟。结果数据保存老出问题，折腾了我整整两周。嗯，从那以后，我对Eep模块就格外上心。

Eep模块到底是干什么的？

说白了，Eep模块就是AUTOSAR架构里负责「假装」你有一块EEPROM的软件层。它底层操作的是Flash，但对外提供的是EEPROM的接口——按字节读写、掉电不丢失。

你想想看，真正的EEPROM芯片贵啊，而且容量小。用Flash模拟，成本低、容量大，但麻烦的是Flash有擦写寿命限制，而且不能像EEPROM那样按字节擦除。Eep模块就是来解决这些矛盾的。

核心功能一句话总结：Eep模块把Flash包装成EEPROM，让上层应用感觉不到底层是Flash。

虚拟EEPROM的实现原理

虚拟EEPROM，其实就是用一块Flash区域来模拟EEPROM的行为。我习惯把这块Flash区域叫做「Eep虚拟区」。它的工作方式是这样的：

  • 数据存储：每次写数据时，不是覆盖旧数据，而是在新的位置写入新数据。旧数据标记为「无效」。
  • 垃圾回收：当虚拟区快写满时，把有效数据搬到一个新块，然后擦除旧块。
  • 地址映射：维护一个逻辑地址到物理地址的映射表，上层读写时通过映射找到实际位置。

我记得第一次看这个设计时，觉得挺绕的。但后来想明白了——这不就是数据库里「日志结构合并树」的简化版嘛！

/* 虚拟EEPROM的地址映射示意 */
typedef struct {
    uint16 LogicalAddress;   /* 逻辑地址，上层看到的 */
    uint16 PhysicalAddress;  /* 物理地址，Flash上的实际位置 */
    uint8  Status;           /* 0x00: 有效, 0xFF: 无效 */
} Eep_AddressMapEntry;

/* 写操作流程（简化版） */
void Eep_Write(uint16 LogicalAddr, uint8* Data, uint16 Length) {
    /* 1. 找空闲物理地址 */
    uint16 PhysAddr = FindFreeSlot(Length);
    
    /* 2. 写数据到Flash */
    Flash_Write(PhysAddr, Data, Length);
    
    /* 3. 更新映射表 */
    UpdateMapping(LogicalAddr, PhysAddr);
    
    /* 4. 标记旧数据为无效 */
    InvalidateOldData(LogicalAddr);
}

磨损均衡策略——让Flash活得更久

Flash的擦写次数有限，一般也就1万到10万次。如果某个地址频繁被写，那块区域很快就报废了。磨损均衡就是要把写操作均匀分布到整个虚拟区。

我常用的策略有三种：

策略名称 原理 适用场景
静态磨损均衡 所有物理块轮流使用，写操作循环分配 数据量小、写入频繁
动态磨损均衡 根据每个块的擦写次数动态选择 数据量大、写入不规律
混合策略 静态为主，动态为辅 大多数量产项目

我的经验：在EB Tresos里配置Eep模块时，有个参数叫「EepVirtualPageSize」。我建议设成Flash物理扇区大小的整数倍。比如Flash扇区是4KB，那虚拟页就设成4KB或8KB。这样垃圾回收时效率最高。

数据保存与恢复——别让客户数据丢了

数据保存，说白了就是保证写进去的数据在掉电后还能读出来。但Flash写入过程中如果突然掉电，数据可能只写了一半。这就是所谓的「掉电安全」问题。

我一般用两种方法：

  • 双缓冲法：数据写两份，一份主数据，一份备份。读的时候先读主数据，如果校验失败就读备份。
  • 日志法：每次写操作先记日志，写完后再更新数据区。掉电后根据日志恢复。

数据恢复呢，就是上电后把虚拟区里的有效数据重新整理好。流程大概是：

  1. 扫描整个虚拟区，找出所有有效数据
  2. 重建地址映射表
  3. 如果发现数据不一致，用备份或日志恢复
  4. 必要时触发一次垃圾回收

注意：我曾经在一个项目里遇到过，上电恢复时映射表重建失败，导致所有数据都丢了。后来发现是映射表本身没有做校验。所以，映射表一定要加CRC校验，而且校验值要单独存。

EB Tresos里的配置要点

在EB Tresos里配Eep模块，有几个参数我每次都会仔细检查：

  • EepJobEndNotification：操作完成后的回调函数。我习惯在这里加一个信号量，通知任务层操作完成。
  • EepWriteVerify：写完后是否回读验证。建议打开，虽然慢一点，但安全。
  • EepImmediateData：是否支持立即写。如果上层要求实时保存，这个必须开。

避坑指南

最后分享几个我踩过的坑：

  • 坑一：忘记考虑Flash的擦除时间。有些Flash擦一个扇区要几百毫秒，如果上层任务有实时性要求，必须用异步方式。
  • 坑二：磨损均衡算法太复杂。我见过有人用「最空闲块优先」策略，结果每次写都要扫描所有块，性能惨不忍睹。
  • 坑三：数据恢复时没考虑「写了一半」的情况。掉电后数据区可能处于不一致状态，恢复逻辑必须能处理这种中间态。
