第九章 中断控制器（Icu）：按键中断实战

各位同学，欢迎来到中断控制器这一章。说实话，Icu 这个模块在 MCAL 里看起来不起眼，但用不好真的会出大问题。我早期做项目时，就因为边沿检测配置错了，导致按键按下去没反应，查了两天才找到原因。今天咱们就把 Icu 彻底讲透。

9.1 Icu 模块到底在干什么？

Icu 的全称是 Input Capture Unit，说白了就是输入捕获单元。它的核心任务就两个：检测信号边沿、测量信号时间。

你想想看，一个按键按下去，硬件上就是电平从高变低（或者从低变高）。Icu 就是负责捕捉这个变化的瞬间。它不像 Gpt 那样自己产生时钟，而是被动地等待外部信号变化。

Icu 的三大核心功能：

  • 信号边沿检测：检测上升沿、下降沿、双边沿
  • 信号时间测量：测量脉冲宽度、周期、占空比
  • 唤醒功能：在低功耗模式下通过信号边沿唤醒 ECU

我个人习惯把 Icu 分成两类应用场景：一类是简单的按键检测，只需要知道有没有按下；另一类是复杂的信号测量，比如 PWM 输入捕获。咱们这节课重点讲按键检测，因为这是最常用也最容易出错的。

9.2 信号边沿检测：上升沿还是下降沿？

这里有个关键问题：按键按下时，到底检测哪个边沿？

我见过很多新手直接配成下降沿触发，结果发现按键松开时也会触发一次。为什么会这样？因为机械按键存在抖动现象。按下时会有多次弹跳，松开时同样有。

按键状态 理想波形 实际波形（含抖动）
按下 高→低（下降沿） 高→低→高→低→...→稳定低
松开 低→高（上升沿） 低→高→低→高→...→稳定高

避坑指南：我曾经在一个量产项目中，直接用了下降沿检测按键按下，结果每按一次，中断服务程序被触发了七八次。后来加了 20ms 的软件消抖才解决。记住：硬件边沿检测 + 软件消抖，缺一不可。

我的建议是：按键按下用下降沿检测，然后在中断服务程序里启动一个定时器做消抖。消抖时间一般 10-20ms，具体看按键的机械特性。

9.3 中断优先级配置：别让按键抢了 CAN 的活

中断优先级配置，说白了就是给中断排个队。谁先来谁后到，谁可以打断谁。

在 EB Tresos 里配置 Icu 中断优先级时，我一般遵循这个原则：

  • 高优先级：CAN 接收中断、定时器中断（周期短、实时性要求高）
  • 中等优先级：SPI 传输完成中断、ADC 转换完成中断
  • 低优先级：按键中断、外部唤醒中断（允许被其他中断打断）

你想想看，如果按键中断优先级设得比 CAN 还高，那 CAN 报文来了都得等按键处理完。万一按键按下时正好有重要报文丢失，后果很严重。

小技巧：在 EB Tresos 的 Icu 配置界面里，每个通道都可以单独设置优先级。我习惯把按键中断优先级设为 3（0 最高，7 最低），这样既不会被频繁打断，也不会影响关键通信。

9.4 实战：按键中断配置（EB Tresos 操作）

好，理论讲完了，咱们直接上手配置。假设硬件连接是：按键接在 Port 的 Pin 5 上，对应 Icu 通道 0。

步骤一：配置 Icu 通道

打开 EB Tresos，找到 Icu 模块配置界面：

IcuConfigSet → IcuChannelConfig
  - IcuChannelId: 0
  - IcuActivationCondition: ICU_ACTIVATION_CONDITION_FALLING_EDGE
  - IcuMeasurementMode: ICU_MEASUREMENT_MODE_SIGNAL_EDGE_DETECT
  - IcuSignalProcessing: ICU_SIGNAL_PROCESSING_IMMEDIATE
  - IcuWakeupCapability: FALSE

这里要注意 IcuSignalProcessing 这个参数。我刚开始做的时候选了 ICU_SIGNAL_PROCESSING_FILTERED，结果发现硬件滤波器把正常的按键信号也滤掉了。后来改成 IMMEDIATE，配合软件消抖，效果很好。

步骤二：配置中断

IcuConfigSet → IcuInterruptConfig
  - IcuInterruptId: Icu_Irq_Channel0
  - IcuInterruptPriority: 3
  - IcuInterruptType: ICU_INTERRUPT_TYPE_EDGE
  - IcuInterruptEnable: TRUE

嗯，这里有个坑。有些芯片的 Icu 中断向量是共享的，比如通道 0 和通道 1 共用一个中断号。配置时一定要查芯片手册，确认每个通道的中断向量是否独立。

步骤三：代码实现

配置完成后，在代码里实现中断服务函数：

/* Icu 中断服务函数 */
void Icu_IrqHandler(void)
{
    /* 清除中断标志 */
    Icu_ClearInterrupt(IcuConf_IcuChannelConfig_IcuChannelConfig_0);
    
    /* 启动软件消抖定时器 */
    SchM_Enter_Icu_Exit();
    Gpt_StartTimer(GptConf_GptChannelConfig_GptChannelConfig_0, 20000); /* 20ms */
    SchM_Exit_Icu_Exit();
}

/* 定时器回调函数（消抖完成） */
void Gpt_Callback(void)
{
    uint8 KeyState;
    
    /* 读取按键当前电平 */
    KeyState = Dio_ReadChannel(DioConf_DioChannel_DioChannel_5);
    
    if (KeyState == LOW)
    {
        /* 按键确实按下，执行按键处理 */
        Key_Process();
    }
}

关键点：中断服务函数里只做最少的操作——清除标志、启动定时器。真正的按键处理放在定时器回调里。这样可以避免中断嵌套导致的问题。

9.5 常见问题与调试技巧

做 Icu 配置时，我遇到过几个典型问题，分享给大家：

  • 问题一：中断不触发——检查 GPIO 复用功能是否配置正确。有些芯片的 Icu 输入需要先配置 Port 的备用功能。
  • 问题二：中断触发多次——硬件抖动导致。加消抖，或者用 ICU_SIGNAL_PROCESSING_FILTERED 配合硬件滤波器。
  • 问题三：中断响应太慢——优先级设低了。检查是否有更高优先级的中断长时间占用 CPU。

我记得有一次调试，按键中断死活不触发。用示波器量引脚，波形没问题。查了半天，发现是 Port 模块的 PortPinDirection 配成了输出。你说冤不冤？所以配置 Icu 之前，一定先确认 Port 的引脚方向是输入。

9.6 小结

这一章咱们讲了 Icu 的核心功能、边沿检测的选择、中断优先级的配置思路，还手把手做了按键中断的实战配置。说白了，Icu 就是个信号捕捉器，用好了能让你的系统响应灵敏，用不好就是各种莫名其妙的问题。
