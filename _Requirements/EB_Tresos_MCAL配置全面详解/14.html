第十四章：控制器局域网（CAN）：CAN模块概述、位时序配置、消息对象配置、硬件过滤与FIFO

好，咱们今天聊聊CAN。说实话，在汽车电子这个圈子里混，CAN总线是绕不开的坎。我最早接触CAN是在一个车身控制项目上，那时候刚入行，被位时序折腾得够呛。后来慢慢摸透了，才发现这东西其实挺有意思的。

CAN模块在AUTOSAR架构里属于MCAL层，说白了就是直接跟硬件寄存器打交道的部分。EB Tresos里配置CAN，核心就三件事：时序要对、消息要准、过滤要狠。咱们一个一个说。

14.1 CAN模块概述

CAN控制器在芯片内部，负责把CPU要发的数据打包成CAN帧，或者把收到的CAN帧解析成数据。我习惯把CAN模块想象成一个智能邮局——你只管把信（数据）交给它，它自己会处理地址、校验、重传这些破事。

在EB Tresos里，CAN模块的配置项很多，但别慌。核心配置就这几个：

  • CanController：每个CAN控制器实例，比如Can0、Can1
  • CanHardwareObject：硬件对象，对应消息对象或FIFO
  • CanFilterMask：过滤掩码，决定哪些报文能进来

嗯，这里要注意：一个CAN控制器可以配多个硬件对象。每个硬件对象要么是消息对象（Message Object），要么是FIFO。这个后面细说。

核心要点：CAN模块的配置，本质上是告诉硬件：你该用多快的速度跑，哪些报文你要收，收完了放哪里。

14.2 位时序配置

位时序，说白了就是决定CAN总线跑多快。CAN 2.0标准下，速率最高1Mbps。但实际项目中，125kbps、250kbps、500kbps更常见。为什么？因为线束长了、节点多了，速率太高容易出错。

位时序由几个参数决定：

参数 说明 我的经验
Baudrate 波特率，比如500kbps 别超过1M，除非你线缆质量极好
Sync_Seg 同步段，固定1个Tq 这个不用动，硬件自己处理
Prop_Seg 传播段，补偿物理延迟 线越长，这个值越大
Phase_Seg1 相位段1 跟采样点位置有关
Phase_Seg2 相位段2 决定了采样点靠前还是靠后
SJW 同步跳转宽度 一般设1~2，太大容易误同步

你想想看，这些参数组合起来，就决定了采样点的位置。我个人习惯把采样点设在75%~85%之间。太靠前，容易采到信号抖动；太靠后，又可能错过有效电平。

小技巧：EB Tresos里有个位时序计算器，你输入目标波特率和时钟频率，它会自动算出推荐值。但我建议你算完后，手动验证一下采样点位置。我曾经在一个项目上直接用了工具推荐值，结果总线在高温下频繁出错——后来发现采样点偏了。

举个例子，假设系统时钟40MHz，目标500kbps，我通常会这样配：

// 伪代码示例
CanControllerConfig.Baudrate = 500000;
CanControllerConfig.PropSeg = 2;   // 2 Tq
CanControllerConfig.PhaseSeg1 = 3; // 3 Tq
CanControllerConfig.PhaseSeg2 = 2; // 2 Tq
CanControllerConfig.SJW = 1;       // 1 Tq
// 总Tq数 = 1 + 2 + 3 + 2 = 8 Tq
// 采样点 = (1+2+3)/8 = 75%

嗯，75%的采样点，对于大多数场景都够用了。

14.3 消息对象（Message Object）配置

消息对象，是CAN模块里最灵活的部分。每个消息对象可以独立配置：

  • 方向：发送还是接收
  • ID：标准帧11位，扩展帧29位
  • 数据长度：DLC，0~8字节
  • 中断使能：收完或发完要不要进中断

我记得第一次配消息对象时，犯了个低级错误——把两个接收对象配成了同一个ID。结果总线上一有那个ID的报文，两个对象同时触发，中断处理乱成一锅粥。后来我养成了习惯：每个消息对象的ID必须唯一，除非你故意用FIFO。

在EB Tresos里，消息对象的配置界面长这样：

CanHardwareObject_0:
  - Type: MessageObject
  - Direction: Receive
  - CanId: 0x123
  - IdType: STANDARD
  - Dlc: 8
  - Interrupt: ENABLED

这里有个坑：DLC配错了，数据可能被截断。比如你配了DLC=4，但对方发了8字节，你只能收到前4个。我曾经排查过一个故障，对方发了8字节，我这边只收到4个，查了半天才发现是DLC没改。

警告：消息对象的数量是有限的！不同芯片不一样，有的只有16个，有的64个。别一股脑全配成消息对象，省着点用。不够了就用FIFO。

14.4 硬件过滤与FIFO

硬件过滤，是CAN模块的看门神。它决定了哪些报文能进到你的消息对象里。过滤靠两个东西：

  • 过滤码（Filter Code）：你想收的ID
  • 掩码（Mask）：哪些位必须匹配，哪些位无所谓

举个例子，你想收ID=0x100~0x10F的报文，可以这样配：

FilterCode = 0x100;
Mask = 0x7F0;  // 高7位必须匹配，低4位无所谓

这样，0x100、0x101、0x102...0x10F都能进来。0x110就进不来，因为高7位变了。

我建议你尽量用掩码来批量过滤，而不是每个ID单独配一个消息对象。这样能省下宝贵的硬件资源。

再说FIFO。FIFO是个好东西，它像一个自动排队机。你配一个FIFO，设置好深度（比如8个槽位），然后所有匹配的报文按顺序往里塞。你只管从FIFO里读就行，不用管哪个报文对应哪个对象。

FIFO适合什么场景？

  • 多ID接收：比如你要收10个不同ID的报文，用FIFO一个对象就够了
  • 高频率报文：比如CAN FD，数据量大，FIFO能防止丢帧
  • 调试阶段：不确定哪些ID会来，FIFO全收了再说

但FIFO也有缺点：你没法给每个报文单独配中断。FIFO满了才触发中断，或者每收到一个报文都触发。前者可能丢帧，后者中断太频繁。我一般折中：FIFO深度设4~8，中断在水印（比如半满）时触发。

实战建议：关键报文（比如刹车、转向）用消息对象，单独配中断。非关键报文（比如车窗、灯光）用FIFO，批量处理。这样既保证了实时性，又节省了资源。

最后说一句，硬件过滤和FIFO是配合使用的。你配了FIFO，也得配过滤掩码，否则所有报文都往FIFO里塞，CPU会被撑死。我曾经见过一个同事，FIFO没配掩码，结果总线上一堆诊断报文全进来了，CPU负载直接飙到90%...嗯，后来他学乖了。

好了，CAN模块的核心配置就这些。下一章咱们聊聊CAN的唤醒和错误处理，那又是另一番天地了。
